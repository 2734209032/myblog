# 1.聊聊vue2和vue3的响应式原理

1.vue2中
在 Vue 实例初始化时，通过遍历 data 对象及其所有属性进行递归，并对对象属性使用 Object.defineProperty 进行 getter/setter 的绑定。当用到某个数据时，在getter函数中进行依赖收集，当数据发生变更时，在setter 函数异步的调用render函数(使用的是发布订阅)，返回虚拟dom，再调用diff函数，对比新旧虚拟dom，进行差异化更新，最后，patch，渲染页面
缺陷是： 1.1 初始化时，需要递归的给每个data中的数据绑定getter和setter，如果数据很多的话，就容易栈溺出，而且页面加载比较慢
         2.2 不能监听对象和数组下标修改属性，或新增属性，解决办法还是有，vue2重写了数组的方法，push put...还有 this.$set()
          语法是 this.$set(target, key, value) 我记得Object.assign()好像也可以 // 利用 Object.assign 动态添加属性
          Object.assign(this.data, {newData: 'new data'})
2.vue3中 响应式系统使用 ES6 Proxy 对象
在创建 reactive 对象时，Vue.js 3.x 使用 new Proxy() 函数来包装原始数据对象（例如 data），然后返回一个新的代理对象。
如果应用程序取回被代理的数据，则代理对象会返回封装的值，同时也会收集读取操作的依赖（即类似于 Vue.js 2.x 的 getter）。
当一个值被修改时，代理对象将通过自身 set 函数来拦截该操作，并触发更新策略。这个过程与 Vue.js 2.x 类似 -- 所有依赖项将会在更新队列中排队并等待下一个 tick 进行更新。
优点:
2.1数组变化检测的性能得到改善，数组的变化可以直接被 Proxy 劫持到。
2.2对象中的属性动态添加/删除问题也得到了解决。Proxy 代理对象可直接监听所有新加或移除属性，并进行相应操作。
2.3Proxy 监听添加或删除属性比 Object.defineProperty() 监听大数据对象变化更高效，因为 Proxy 只针对被新增、修改和删除对象进行响应。
因此，Vue.js 3.x 在响应式系统方面的优化显著提升了 Vue.js 开发的效率和性能，特别是在处理大量数据时更加具有优势。
//总之
//Vue.js 3.0 比 Vue.js 2.x 在性能上的优势主要体现在初始化阶段，不需要递归把子对象定义成响应式。而 Proxy 本身并不比 Object.defineProperty 快，好处是在于可以直接对整个对象劫持，包括对象属性的新增和删除，劣势就是浏览器的兼容性不够好，而且没有合适的 polyfill。

# 2.ref和reactive区别

使用场景不同：
1.ref 主要用于包装基本数据类型（例如数字、字符串等），以便能够通过类似变量的操作直接读取和修改其值；而 reactive 则用于包装复杂数据类型（例如对象、数组等），并将其转换为响应式对象，以便在数据发生变化时自动触发更新。
//注意一点的是ref也能定义复杂数据类型，它的底层是借助了reactive函数
2.对象属性访问方式不同：使用 ref 定义的变量，在模板中需要通过 .value 来获取其实际值，否则会只获取其内部包装对象；而 reactive 对象的内部属性可以直接访问，无需手动添加 .value。
3.内部实现原理不同：ref 通过封装一个普通的 JavaScript 值来实现响应式，并使用了 Object.defineProperty() 方法对其读取和修改进行拦截处理；而 reactive 则是通过 Proxy 来代理整个对象，并在其中拦截了对象上的所有操作，从而实现了对嵌套属性的监听支持。

# 3.vue3的组件通信方式有哪些？和vue2通信的区别

vue3
1.Props：用于父组件向子组件传递数据的主要方式，使用时需要在子组件中声明 props 接收传递过来的值。
2.Emit：用于子组件向父组件传递数据的方法，通过在子组件中使用 $emit() 方法触发事件并将数据传递给父组件。

# 4.vuex和pinia的区别

Vuex 和 Pinia 都是 Vue.js 生态中的状态管理库，并且它们有着不同的特点。
1.数据存储位置不同：
Vuex 存储数据的位置是全局唯一的，适用于大型应用或需要多个页面共享状态的场景。而 Pinia 存储数据的位置是独立于每个组件实例，因此对其他组件的影响非常小，常用于小型和中型应用，以及针对单个组件封装的插件和组件库。
2.数据操作方式不同：
在 Vuex 中，通过 mutations、actions 和 getters 等方式对数据进行操作和获取。相应地，在使用 Vuex 时需要提前定义好 state、mutations、actions 和 getters 等模块，这就导致代码量较大，同时也可能出现命名冲突等问题。
Pinia 中数据操作的方式则更加简单和易于使用。具体而言，Pinia 提供了 useStore() 函数来创建 store 实例并访问其中的 state、actions、getters 和 mutations 等属性，这种方式可以减少代码复杂度和提高开发效率。
3.底层实现原理不同：
Vuex 的底层使用了 Vuex.Store 类来创建一个全局的 Store 实例，Store 实例包含了 State、Mutation、Action、Getter 等对象，State 用于管理状态数据，Mutation 用于修改状态数据，Mutation 必须是同步的修改，Action 用于异步修改状态数据，Getter 用于获取状态数据。
而 Pinia 基于 Vue.js 提供的响应式系统，将 state、action 和 getter 统一存储在一个 reactive 对象中，并使用 computed() 函数来计算 getter。这种方式简化了状态管理的实现，同时也提供了更好的性能和开发体验。
综上所述，Vuex 更适合大型应用，Pinia 更适合小型和中型应用。如果需要引入全局状态管理，使用 Vuex 特别有优势，如果只是针对单个页面或组件进行状态管理，建议选择 Pinia，它的使用体验更好，代码更加简洁易懂

# 6.vue3中wacth和wacthEffect的区别

watch 和 watchEffect 都能监听响应式数据的变化，不同的是它们监听数据变化的方式不同。
watch 会明确监听某一个响应数据，而 watchEffect 则是隐式的监听回调函数中响应数据(只要依赖项变了，就会触发)。
watch 在响应数据初始化时是不会执行回调函数的，watchEffect 在响应数据初始化时就会立即执行回调函数

  const to = setTimeout(() => {

  }, 500)
  onCleanup(() => {
    // 下一次变化时，清除上一次定时器(下一次变化来之前，执行一次)
    clearTimeout(to)
  })
}，

# 7.遇到的状态码有哪些，什么情况遇到的

1. //1XX：信息性状态码
100 Continue：服务器已经接收到请求头，并且将继续处理这个请求。
101 Switching Protocols：客户端正在请求服务器切换协议，比如从 HTTP 1.1 切换到 WebSocket。
2. //2XX：成功状态码
200 OK：请求成功，响应中包含所请求的资源。
201 Created：在服务器上成功创建了新资源。
204 No Content：服务端已经接收到并成功处理了请求，但不会返回消息体内容。
3. //3XX：重定向状态码
301 Moved Permanently：所请求的资源已经被永久移动到其他位置（URL），响应中包含对新 URL 的引用。
302 Found：所请求的资源暂时被移动到其他位置（URL），响应中包含对新 URL 的引用。
304 Not Modified：在使用缓存机制的情况下，表示客户端的缓存资源是最新的，无需重新获取。夹带的数据不会发生改变，由于客户端有该文件的缓存，也就不需要从服务端重新pull该文件，而是直接使用本地缓存。
4. //4XX：客户端错误状态码
400 Bad Request：请求参数有误或语法错误，服务器无法理解此请求。
401 Unauthorized：请求未认证，需要提供身份验证凭据才能访问所请求的资源。
403 Forbidden：请求被服务器拒绝，没有足够的权限访问所请求的资源。
404 Not Found：所请求的资源不存在于服务器上。
422 Unprocessable Entity：客户端提交了格式正确但无法处理的实体（例如，重复的用户名）。
5. //5XX：服务器错误状态码
500 Internal Server Error：服务器遇到了意外情况，且无法处理这个请求。
502 Bad Gateway：代理服务器从上游服务器接收到一个非法的响应。
503 Service Unavailable：服务不可用，一般是由于负载过大或者正在进行服务升级维护等原因导致。
504 Gateway Timeout：代理服务器在规定时间内没有从上游服务器收到请求，通常是由于高并发压力过大导致。

# 8.你git是怎么实现版本管理的

Git通过记录文件的变化来实现版本管理，具体实现过程如下：
1.在本地工作目录中新建或修改文件。
2.使用git add命令添加文件到Git暂存区（Index）。
3.使用git commit命令将暂存区的所有更改保存到Git仓库的一个新版本中，同时记录这个版本的作者、时间等信息。每个新版本都包含了之前所有版本中所有文件内容的快照，因此每个版本都是完整的副本。
4.如果需要查看历史版本，可以使用git log命令列出所有提交记录，每个记录包含作者、提交日期、提交消息和SHA-1校验和等信息。可以使用SHA-1校验和查询特定版本。
5.如果需要回退到某个历史版本，可以使用git checkout或git reset命令切换到指定版本。git checkout会创建一个分离头指针，不建议在此基础上进行编辑，而git reset则可以撤销一个或多个提交记录并更新HEAD分支指针。
//在使用Git进行版本管理时，通常需要遵循以下几个基本原则：
1.频繁地提交：尽量将一些小的修改提交成多个较小的版本。
2.持续记录版本：只要更改了文件就及时提交，不要将太多未提交文件积累。
3.使用分支：如果需要开发新功能或修复bug，则应在一个新分支上进行，不要在主分支上直接修改。
4.定期合并：当开发完成后，在开发分支上使用git merge命令将更改合并到主分支

# 9.网络劫持

网络劫持是指黑客或不法分子在网络中篡改、拦截或模拟网络流量，使得用户访问到的网站或服务与其期望不符。网络劫持可以用于信息窃取、钓鱼攻击、广告投放等目的。
常见的网络劫持方式包括：

1.DNS劫持：将域名解析结果篡改为攻击者的IP地址，使得用户访问的网站实际上是攻击者伪造的网站。
2.HTTP劫持：在数据传输过程中中间人攻击，修改HTTP报文，使得用户看到的内容不是原始的响应内容。
3.SSL Striping: 在HTTPS请求时，欺骗用户使用HTTP连接，然后将请求转发到HTTPS服务器，并获取加密内容。
4.ARP劫持：在局域网中欺骗其他计算机，让它们把数据发给攻击者而非实际的目标机器。
5.WiFi重定向：伪装WiFi热点，让受害者接入该热点并进入假网站。
为了防止网络劫持，我们需要采取以下措施：

1.使用HTTPS协议：通过使用HTTPS协议来加密数据传输，可以有效防止SSL Striping和HTTP劫持。
2.确认证书正确性：确保网站提供的SSL数字证书是由可信机构颁发的，并且可以验证其合法性。
3.避免连接不安全的WiFi：尽量避免在公共场所或未经身份认证的WiFi上进行敏感操作。
4.安装防护软件：安装有效的网络安全软件来检测并拦截可能存在的攻击行为。
6.确保DNS服务器安全：使用可信赖的DNS服务器，并加强对本机和路由器DNS配置的管理。
总之，要保障自己的信息安全，需要做好网络安全方面的知识储备和相应的预防措施。

# 10.项目从0到1的开发流程

首先，从0到1的项目开发需要进行详细的需求分析和技术可行性研究。接下来，可以按照以下步骤进行项目实现：
1.编写项目计划书和开发文档，明确项目的目标、功能需求、技术选型、开发流程、测试方案等。
2.进行系统设计与架构设计，根据实际需求进行技术选型，包括数据库、Web框架、服务器等。
3.建立代码仓库和版本控制，选择常用的Git或者SVN等工具进行管理和协同开发。
4.根据需求和设计，在本地建立项目文件，并进行静态页面设计，采用HTML、CSS、JS等技术，初步形成项目框架。
5.写出后台API接口，实现前端页面渲染所需的数据获取，以及提供必要的服务接口等。
6.在前端与后台之间建立数据交互通道，通过AJAX或WebSocket等技术，实现前后端数据交互，并对数据进行验证和安全加密等。
7.完成业务逻辑的编写，实现前端页面与后台API的交互，包括表单提交、搜索、筛选、排序、分页、数据展示等。
8.系统性能优化，包括网站速度优化、CDN以及缓存技术、并发性和负载均衡、数据备份与恢复等。
9.测试和调试，进行单元测试和功能测试，并进行性能测试和安全测试，确保系统符合需求和规格要求。
10.部署和上线，搭建服务器环境并进行部署，包括数据库配置、文件上传、域名设置等。同时，确保系统的安全性，防止系统漏洞被利用。

# 12.对meta标签的理解

<meta>标签是HTML中的一种特殊标签，用于描述网页文档的各种元信息和属性。主要作用包括以下几个方面。
1.定义网页文档的编码格式：通过设置charset属性来指定文档采用的字符集编码，例如：<meta charset="UTF-8">
2.引入外部资源：例如引入CSS样式表或者JavaScript脚本等，在<head>标签内添加<link>或<script>标签即可。
3.描述网页文档的相关信息：使用<meta name="keywords"> 和<meta name="description">等标签来定义页面的关键词和描述信息，以便搜索引擎正确抓取并展示网页信息。
4.控制缓存策略：通过设置<meta http-equiv="expires">或者<meta http-equiv="cache-control">等属性来定义页面是否能被缓存，以及缓存有效期。
5.支持移动设备：例如设置<meta name="viewport">来为不同尺寸和不同设备自适应地调整页面布局和风格。
总之，<meta>标签可以帮助网页提供更多的信息，同时支持页面的优化与最佳实践，也可以使页面适应更多的场景需求，比如渲染速度、SEO优化、设备兼容性等。

# 15.说说页面布局吧，弹性布局的话

//，有没有去了解过具体实现的原理，比如用flex实现盒子垂直水平居中，其中用到的 displav: flex;justify-content: center;align-items: center;具体是什么原理实现的
.container {
  display: flex;

具体实现的原理如下：

4.align-items: center;属性用于控制伸缩项在交叉轴方向上的对齐方式，把它们垂直居中对齐。

因此，当display: flex;、justify-content: center;和align-items: center;属性被应用于一个父容器上时，其中所有的子元素都会被居中对齐。通过灵活运用这些CSS属性，我们可以轻松实现各种复杂的页面布局效果。

# 16.说一下小程序优化

小程序的优化主要包括以下几个方面：
1.减小小程序包体积：可以使用分包、压缩、懒加载等技术，减少小程序包体积，加快小程序启动速度。
2.减少 HTTP 请求数量：减少网络请求可以提高页面加载速度。可以通过合并 JS 文件或者合并 CSS 文件，或者使用 WebFont 等技术，将多个请求合并为一个 HTTP 请求。
3.减少重排和重绘：频繁地对 DOM 进行操作会导致浏览器不断执行重排和重绘的操作，降低小程序性能。针对此问题可以采用避免使用 table 布局、避免使用多层嵌套的 DOM 结构、避免频繁修改样式等技术方法来减少反复渲染的次数。
4.合理使用 Storage：Storage 是一种本地存储机制，如果程序中不断往 Storage 中写入数据会带来性能上的损失，因此应该限制 Storage 的使用，尽可能使用内存变量。
5.少使用定时器：使用过多定时器会影响小程序的性能。为了避免性能问题，建议使用 requestAnimationFrame 替代 setTimeout / setInterval 来达到页面刷新的目的。
6.减少渲染，减轻渲染负担：减少不必要的数据更新，可以降低小程序顶层的渲染数量；通过点进进行自定义组件化开发，尽量减轻微信渲染引擎渲染次数。

总之，小程序优化需要从应用的各个方面进行考量和实践，针对性地采取相应的技术手段和策略。

# 20.在列表渲染中，在哪些情况下可以使用数组下标作为key,为什么?

1.数组中的项目没有唯一ID，并且项目之间没有明显的排序关系，而只是一些项的集合。
2.列表是静态的，不会改变顺序或动态添加/移除项目。
这两种情况下可以使用数组索引作为key，因为这样做可以提高列表渲染的效率。由于每一个项目都有自己独特的key，Vue就不需要花时间去比较它们的值来检测它们是否发生了变化，从而优化性能。但是如果列表具有顺序或者是动态变化的，则不应该使用数组的索引作为key，因为这可能会导致未预期的行为和性能问题。

      const img = document.querySelector("img");
      // 有了画板
      const canvas = document.querySelector("canvas");
      // 画笔
      const ctx = canvas.getContext("2d");
      // 画板位置大小
      const rect = canvas.getBoundingClientRect();
      console.log(rect);
      // 鼠标按下，开始，鼠标移动，画线，鼠标抬起，结束
      window.addEventListener("mousedown", (e) => {
        // 鼠标是不是按在画板上 - 命中检测
        if (
          e.clientX < rect.x ||
          e.clientX > rect.x + rect.width ||
          e.clientY < rect.y ||
          e.clientY > rect.y + rect.height
        ) {
          return;
        }
        // 在方块内部
        // 鼠标移动就画画
        // x,y 相对于 canvas
        let x = e.clientX - rect.x;
        let y = e.clientY - rect.y;
        const draw = (ev) => {
          let nextX = ev.clientX - rect.x;
          let nextY = ev.clientY - rect.y;
          // 开始画画
          ctx.beginPath(); // 我要开始描线了
          ctx.moveTo(x, y); // 起点
          ctx.lineWidth = 10;
          ctx.lineTo(nextX, nextY); // 终点
          ctx.strokeStyle = "red";
          ctx.stroke(); // 描线
          // 更新起点
          x = nextX;
          y = nextY;
        };
        window.addEventListener("mousemove", draw);
        // 鼠标抬起
        window.addEventListener("mouseup", () => {
          window.removeEventListener("mousemove", draw);
          window.removeEventListener("mouseup", draw);
        });
      });
      button.addEventListener("click", () => {
        // 生成图片
        // img.src = canvas.toDataURL(); // 生成 base64 图片

          img.src = URL.createObjectURL(file);
          // 清空画板
          ctx.clearRect(0, 0, canvas.width, canvas.height);
        }, "image/png");
      });

# 25.v-show和v-if的了解?优先级?在vue2和vue3中

v-show和v-if都是Vue.js中用于条件渲染的指令，但它们有一些区别。

v-if是惰性的，即如果它的表达式在初始渲染时为false，则会跳过整个元素的渲染。如果表达式在运行时变为true，则插入该元素。如果它在运行时变为false，则将此元素从DOM中删除。由于v-if的行为类似于将元素添加到DOM中或从DOM中删除，因此它具有更高的切换开销，适合用于不经常发生改变的情况。

相反，v-show只是简单地控制元素的CSS属性display的值。它没有像v-if那样的惰性，不管初始值是什么，都始终在页面上渲染该元素。因此，v-show在需要经常切换显示/隐藏状态的情况下表现更好。

在Vue 2和Vue 3中，v-if具有更高的换算优先级。这意味着，如果您在同一元素上同时使用了v-if和v-show指令，则v-if将优先于v-show进行计算，因此v-show将总是忽略。当然，在实际编码过程中，应避免在同一元素上同时混合使用v-if和v-show以避免潜在的逻辑错误。

# 26.v-for和v-if的了解?优先级?在vue2和vue3中

1.在Vue2中，v-for指令的优先级高于v-if指令的优先级。也就是说，如果v-for和v-if同时出现在同一标签上，v-for会先执行，并且优先级高于v-if。
2.而在Vue3中，v-if指令的优先级高于v-for指令的优先级。这意味着，如果v-if和v-for同时存在于同一元素上，则v-if将优先于v-for执行。这是因为，在Vue3的编译器中，v-if指令被编译成一个条件渲染函数，而不是像Vue2那样在每次迭代时都进行条件检查。

# 30.fetch/xhr 有什么区别？和 axios 有什么区别？

fetch 两段 then
fetch(url, { method, headers: {}, body }).then(res=>status, res.json()).then(result)
fetch 调用 url，请求方法，头部设置，请求体设置
第一段then，获取状态码，然后配置处理方式（json，text，blob）
第二段then，处理结果
xhr 有五步
1. const xhr = new XMLHttpRequest()
2. xhr.setHeaders()
3. xhr.open(方法, url)
4. xhr.send()
5. xhr.onreadystatechange 状态为 4，状态码为 200 - 500 时，处理结果
axios
1. 配置 axios 实例（timeout，基地址，结果处理方式）
2. 配置请求拦截器
3. axios.get
区别：
1. axios 是第三方库，基于 xhr，fetch/xhr 是浏览器原生的

# 32.异步组件

等待异步组件时渲染一些额外的内容，让应用有更好的用户体验
如果平时我们直接引入组件的话，app组件会等所有的组件加载完，才会展示页面，当网速较低时，我们可以异步引入组件，在配合Suspense，让app组件先渲染，再在Suspense中等待组件渲染时提供一些额外的内容
1.异步引入组件
import {defineAsyncComponent} from 'vue'
const Child = defineAsyncComponent(()=>import('./components/Child.vue')) //要注册的，省略了
使用Suspense包裹组件，并配置好default与fallback
<template>
<div class="div">
 <Suspense> //原理就是插槽，名字不能改，因为底层给slot命名是这样的
  <template v-slot:default> //default是我们需要展示组件的地方
    <Child></Child>
  </template>
  <template v-slot:fallback>  //fallback是我们在网速加载很慢时，等待组件加载时，展示的内容
    <h3>加载中....</h3>
  </template>
 </Suspense>
</div>
</template>

# 33.git，commit之后没有推送，又写了代码怎么合并这两次本地的代码

1.使用git stash命令将当前未提交的代码保存到工作区堆栈中。
2.使用git pull命令拉取远程仓库的最新代码，并合并到本地分支中。这一步可能导致冲突，请根据提示逐个解决冲突。
3.使用git stash pop命令从堆栈中恢复之前保存的未提交的代码。
4.使用git add和git commit命令将本地修改提交到本地仓库中。
5.最后使用git push将本地仓库的改动推送到远程仓库。

# 34.你有一个页面，有很多组件，很多图片，用户加载很卡，怎么优化，除了懒加载还有什么方式

1.图片压缩
图片是web页面中最耗费带宽和加载时间的元素之一。如果您的页面中包含大量未经过压缩的图像，建议使用图像压缩工具（如TinyPNG.com）将其压缩，来减小图像文件的大小，并通过样式设置正确的图像尺寸以提高响应速度。
2.使用CDN加速
使用内容分发网络（CDN）有效地将网站的静态内容（如CSS、JavaScript、图片）部署到全球范围内的服务器上，以便从距离最近的服务器快速交付这些资产。这样可以更快地下载或缓存页面的静态资源，并使您的网站响应时间更快。
3.代码分割
将代码分割成更小的块后，只加载当前需要的部分，而不是加载整个应用程序代码。在Vue.js中，此功能可以通过将应用程序拆分为可重用的组件并使用Webpack等构建工具对其进行自动处理实现。
4.SSR（服务端渲染）
使用服务器端渲染（SSR）技术将应用程序的初始渲染移动到服务器上，生成HTML、CSS和JS，并在最初的请求时将其发送回浏览器。这样可以更快地在用户浏览器中看到应用程序的展示内容。
5.减少 HTTP 请求数
将 CSS 和 JavaScript 压缩并集成到一个文件中，并减少页面的外部资源请求，从而能够显著地提高网站的加载速度。此外，在实际设计和创作中请尽可能减少 DOM 元素、HTTP 请求次数以及 try-catch 等引擎处理硬件优化。

这些方法都是常见且有效的优化技巧，可以帮助您改善页面加载卡顿问题，提高网站的性能和用户体验。

# 36.vue3的性能优化的方面体现在哪

Vue 3 的性能优化主要体现在以下几个方面：

1.更快的渲染速度：Vue 3 引入了新的编译器，可以生成更少的运行时代码，从而提高了应用程序的启动时间和运行时性能。
2.更小的包大小：Vue 3 将库的大小缩小了近 40%，从而加快了加载速度，并降低了运行时的内存占用。
3.更好的响应能力：Vue 3 改进了其响应式系统，采用 Proxy 监听数据变化，替代了旧版的 Object.defineProperty，从而更好地支持 TypeScript 和使用 Composition API 进行组件开发。
4.更高效的组件更新：Vue 3 引入了静态根节点优化和基于 Proxy 的依赖追踪，提高了组件更新的效率，使得组件的更新速度更快。
5.更好的虚拟 DOM：Vue 3 的虚拟 DOM 采用了更高效的算法，可以减少大量冗余的操作，同时还可以更好地支持 Fragment 和 Teleport 等新特性
